# Skill: frontend-design（フロントエンドデザイン品質）

## 目的
UIタスク実装時に「AIっぽい汎用デザイン」を避け、目的に応じた個性的で完成度の高いインターフェースを作る。
コード前にデザイン方向性を定め、実装後にスクリーンショットで検証する。

## 前提条件
- 対応するスペックが `specs/` に存在
- 変更対象にUI関連ファイル（*.tsx, *.vue, *.css, *.scss等）が含まれる
- Playwright MCPが利用可能（スクリーンショット検証時）

## 手順

### Step 1: デザインシンキング
コードを書く前に、以下の4つの観点を明確化する。

#### 1-1. 目的とオーディエンス
- この画面/コンポーネントは誰のために何を解決するか
- ユーザーが最初に目にする要素は何か
- 主要なユーザーアクションは何か

#### 1-2. トーン/方向性の選択
以下から最も適切なトーンを選択（または独自に定義）:

| トーン | 特徴 | 適するケース |
|--------|------|--------------|
| ブルータルミニマル | 大胆なタイポグラフィ、モノクロ、余白を活かす | テック系、ポートフォリオ |
| マキシマリスト | 密度が高く装飾的、複数の視覚レイヤー | エンタメ、クリエイティブ |
| エディトリアル | 雑誌的レイアウト、活字と画像の調和 | メディア、ブログ |
| ソフト・パステル | 柔らかい色調、丸み、優しいモーション | ヘルスケア、教育 |
| インダストリアル | ダーク、構造的、情報密度が高い | ダッシュボード、開発ツール |
| レトロ/ノスタルジック | 時代感のあるフォント・テクスチャ | ブランディング、特設サイト |

#### 1-3. 技術的制約
- 使用フレームワーク/ライブラリ
- レスポンシブ要件
- パフォーマンス制約（初期表示速度等）
- アクセシビリティ要件

#### 1-4. 差別化ポイント
- 同種のサービスと比べて何が視覚的に異なるか
- 記憶に残る要素は何か

**出力**: デザイン決定書（上記4項目の回答をまとめたもの）をユーザーに提示し承認を得る。

### Step 2: 実装ガイドライン

以下の5つのチェックリストに従って実装する。

#### 2-1. タイポグラフィ
- [ ] 汎用フォント（Inter, Roboto, Open Sans）をそのまま使わず、目的に合ったフォントを選択
- [ ] 見出し用（Display）と本文用（Body）のフォントペアリングが意図的
- [ ] フォントサイズのスケール（ratio）が統一されている
- [ ] 行間・字間がコンテンツ量に応じて調整されている

#### 2-2. カラー＆テーマ
- [ ] 紫グラデーション・青→紫のグラデーションに安易に頼らない
- [ ] 支配色（Dominant）・補助色（Secondary）・アクセント色が明確
- [ ] ダーク/ライトモード対応時、単なる色反転ではなく各モードで最適化
- [ ] カラーコントラスト比がWCAG AA基準（4.5:1以上）を満たす

#### 2-3. モーション
- [ ] ページ読み込み時の統一されたアニメーション（バラバラの微動より効果的）
- [ ] ホバー・フォーカスの反応が一貫している
- [ ] `prefers-reduced-motion` に対応
- [ ] アニメーション時間は200-500msの範囲（体感速度の最適ゾーン）

#### 2-4. 空間構成
- [ ] 対称・均等配置のみに頼らず、非対称・重なり・対角線の流れを検討
- [ ] グリッドを意図的に「崩す」箇所がある（全てグリッド通りは単調になる）
- [ ] 余白が「埋めるべき空間」ではなく「デザイン要素」として機能している
- [ ] ファーストビュー（Above the fold）で最重要メッセージが伝わる

#### 2-5. ビジュアルディテール
- [ ] テクスチャ・グラデーション・ノイズ等がトーンに合致している
- [ ] アイコン・イラストのスタイルが統一されている
- [ ] シャドウ・ボーダーの使い方が一貫している
- [ ] 装飾がコンテンツの邪魔をしていない

#### アンチパターン（避けるべきもの）
以下は「AIが生成した典型的デザイン」と認識されやすいパターン:
- Inter / Roboto / Open Sans のデフォルト使用
- 紫→青のグラデーション背景
- 均等3カラムカードレイアウト
- 汎用的なヒーローセクション（中央揃え見出し + サブテキスト + CTAボタン）
- Tailwind UIそのままのコンポーネント
- 意味のないアニメーション（フェードイン・スライドアップの乱用）

### Step 3: 複雑さのマッチング
デザイン方向性と実装量の一致を確認:
1. Step 1で決めたトーンの複雑さに対して、実装が過不足ないか
2. ミニマルなトーンなのに過剰な装飾を入れていないか
3. マキシマリストなトーンなのに要素が足りないか

### Step 4: プロダクション品質チェック

#### 4-1. レスポンシブ
- [ ] モバイル（375px）での表示が崩れない
- [ ] タブレット（768px）での表示が自然
- [ ] デスクトップ（1280px以上）で余白が適切
- [ ] フォントサイズがビューポートに応じてスケール

#### 4-2. アクセシビリティ
- [ ] キーボードナビゲーションが機能する
- [ ] フォーカスインジケーターが視認可能
- [ ] 画像にalt属性がある
- [ ] セマンティックHTML（見出し階層、ランドマーク）

#### 4-3. パフォーマンス
- [ ] カスタムフォントの読み込みが最適化されている（`font-display: swap`等）
- [ ] 画像が適切なフォーマット・サイズで提供
- [ ] 不要なCSSが含まれていない
- [ ] アニメーションがGPUアクセラレーション対象（`transform`, `opacity`）

### Step 5: スクリーンショット検証
Playwright MCPを使用してビジュアル検証を実施。

1. 開発サーバーを起動
2. 主要画面のスクリーンショットを取得:
   - デスクトップ（1280x720）
   - モバイル（375x667）
3. スクリーンショットを確認し、以下を検証:
   - Step 1のデザイン決定書と方向性が一致しているか
   - Step 2のチェックリスト項目が視覚的に確認できるか
   - レイアウト崩れ・意図しない表示がないか
4. 問題があれば修正し、再度スクリーンショットで確認

## 判定基準
- **PASS**: Step 2のチェックリスト80%以上クリア & Step 4全パス & スクリーンショット検証OK
- **FAIL**: 上記を満たさない

## 失敗時の対応
1. Step 2のチェックリスト未達 → 該当項目を修正して再チェック
2. Step 4のプロダクション品質未達 → 該当項目を修正
3. スクリーンショットでデザイン決定書との乖離 → デザイン方向性を再確認し修正
4. 3回修正しても解決しない → ユーザーにデザイン方向性の再検討を提案

## 出力
- デザイン決定書（Step 1の結果）
- スクリーンショット（デスクトップ + モバイル）
- チェックリスト結果サマリー
